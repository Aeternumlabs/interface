## The Problem Phase 3 Solves

After Phase 2, a user holding ETH on Ethereum mainnet is fully protected — their hybrid wallet monitors inactivity and recovers assets automatically. But most active crypto users do not live on one chain. They hold ETH on Arbitrum for DeFi, bridge to Base for lower fees, keep BNB on BNB Chain, and hold stablecoins on Polygon.

Each of those positions is currently an orphan from a recovery perspective. If the user becomes incapacitated, the Ethereum mainnet recovery fires and moves their mainnet ETH to the backup address. Their Arbitrum, Base, and BNB positions sit unprotected — subject to the same original problem that Aeternum was built to solve.

Phase 3 closes that gap by expanding the Aeternum hybrid wallet and `AeternumRecoveryManager` to every major EVM-compatible network.

---

## Independent Per-Chain Deployments

Phase 3 does **not** use cross-chain messaging or bridge infrastructure. Every supported network gets its own fully independent deployment:

- Its own `AeternumRecoveryManager` contract
- Its own Chainlink Automation upkeep, funded with the network's native LINK
- Its own rolling cursor registry covering all users on that chain
- Its own `setForwarder()` call, locking execution to the chain-specific Chainlink forwarder

This design is intentional. Cross-chain recovery — where a trigger on Ethereum causes a transfer on Arbitrum — introduces bridge dependency risk, oracle latency, and message-passing attack surfaces. By running independently per chain, each deployment inherits the same trust properties as the Phase 1 Ethereum vault: no cross-chain message can interfere with recovery execution, and a failure on one chain does not block recovery on any other.

Assets on each chain recover independently to the designated backup address **on that same chain**.

## Per-Chain Registration

Users protecting assets across multiple chains register separately on each chain. The registration flow is identical to Phase 2 on each network — connect wallet, designate backup address, set inactivity period.

**The backup address is configured per deployment.** A user can choose a different backup address for each chain, or use the same address everywhere. Common patterns:

- **Same backup everywhere** — simplest setup, one address receives everything across all chains on recovery
- **Chain-specific backups** — different trusted parties per chain, or a chain-specific multisig for each network's asset
- **Hardware wallet as universal backup** — a single Ledger address designated everywhere, receiving all multi-chain assets

**The inactivity period is also per-chain.** A user might want a shorter timer on their active DeFi chains (Base, Arbitrum) and a longer timer on cold-storage chains (Ethereum mainnet). This flexibility is fully supported.

## Activity Synchronisation

In Phase 2, opening the Aeternum app resets the timer on the connected chain. In Phase 3, the app sends activity pings to **all chains where the user has a registered vault** simultaneously, using a batched background signature flow.

This means a user who opens their Aeternum app on any device automatically extends their inactivity deadline across every protected chain — without needing to manually interact on each network individually.

For users who prefer a more hands-on approach, the manual `ping()` function remains available on each chain's contract independently.

## What Does Not Change

The rolling cursor architecture, `MAX_RECOVERY_ATTEMPTS` abandonment handling, stale data safety, and the Chainlink forwarder access control pattern are deployed identically across all chains. Each chain's deployment is a direct copy of the Phase 2 contract, parameterised for that network's Chainlink environment.

The only chain-specific configuration is:
- The Chainlink Automation forwarder address (assigned per-upkeep by Chainlink on each network)
- `CURSOR_ADVANCE_INTERVAL` — tuned per network based on block times and gas economics
- `MIN_INACTIVITY_PERIOD` — may be adjusted per network based on ecosystem expectations
